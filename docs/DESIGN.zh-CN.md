# UI 设计规范

## 1. 定位

这份文档描述的是当前仓库里已经实现出来的 UI 风格，并把这套风格整理成后续工作的默认规范。

它首先是描述性的，其次才是约束性的：

- 记录当前已在成品界面中稳定存在的设计模式。
- 规定后续新增 UI 如何继续与这些模式保持一致。
- 不应被当成另一套理想化产品重设计说明书。

如果当前代码和旧设计描述冲突，以仓库中已经落地的 UI 为准，除非新的设计任务明确要求改变方向。

这个仓库对应的是桌面优先的 AI 客户端。界面应当呈现为克制、实用的工具软件，而不是营销站、视觉实验品或品牌展示页。

## 2. 事实来源优先级

判断设计决策时，按下面的优先级处理：

1. 仓库中当前已落地的 UI
2. 共享 token 和共享 UI 基础组件
3. 本文档
4. 更早的截图、评论和历史描述

具体含义：

- 如果旧文档已经过时，不要反过来要求新工作去迎合它。
- 如果共享 UI 层已经定义了模式，不要在功能组件里重新发明一套本地视觉系统。
- 如果确实需要新模式，应优先沉淀到共享 token 或共享基础组件，而不是直接写成某个功能页专用样式。

## 3. 核心视觉方向

当前 UI 是一套以黑白灰为基础的、克制的、桌面优先 AI 客户端界面。

核心特征：

- 深色模式是主参考模式。
- 浅色模式必须继续支持。
- 整体气质安静、实用、偏工具型。
- 视觉层级主要靠对比、间距和字重建立，而不是靠颜色堆叠。
- 表面层应保持低噪音，阴影和层次感都要克制。

界面整体应该更像“干净的工具软件”，而不是：

- 玻璃拟态展示页
- 高饱和品牌产品页
- 编辑型落地页
- 密集 IDE 式控制面板墙

## 4. 主题与令牌策略

### 4.1 主题模型

- 应用通过根节点上的 `.dark` class 显式切换主题。
- 不应假设 `prefers-color-scheme` 是当前主题的事实来源。
- 深浅主题必须共享同一套结构和组件逻辑。
- 主题切换主要通过 token 完成，而不是通过布局改造完成。

### 4.2 核心语义令牌

当前 token 系统定义在 `web/src/index.css` 中，并通过 Tailwind 主题暴露出来。

核心语义令牌包括：

- `background`
- `foreground`
- `card`
- `card-foreground`
- `muted`
- `muted-foreground`
- `border`
- `accent`
- `accent-foreground`
- `primary`
- `primary-foreground`
- `success`
- `warning`
- `danger`

规则：

- 在引入新全局 token 之前，优先复用现有语义 token。
- 如果现有 token 已经能表达语义，就不要在功能组件中写死局部颜色。
- `success`、`warning`、`danger` 应被视作克制的状态色，而不是品牌主色。

### 4.3 颜色行为

浅色模式：

- 近白背景
- 黑色或接近黑色的前景文本
- 柔和的中性色表面

深色模式：

- 被压缩在灰阶区间内
- 视觉噪音低
- 边框和状态变化克制

深浅模式统一规则：

- 边框保持低对比。
- 主要强调通常来自反差反转，而不是彩色填充。
- 大面积高饱和颜色不属于当前基础视觉语言。

### 4.4 字体令牌

当前字体栈就是后续默认规范：

- 正文：`Geist Variable`，并带 CJK 系统回退
- 标题：与正文相同的字体族
- 等宽：`Geist Mono Variable`，并带标准 monospace 回退

除非未来产品方向明确变化，否则不要引入装饰性展示字体。

### 4.5 圆角、阴影与模糊

当前方向：

- 控件整体圆润但紧凑。
- Pill 胶囊形常用于按钮、小控件和选项行。
- 中等圆角用于卡片、气泡、弹层和次级容器。
- 更大的圆角留给 composer 和设置弹窗这类主要壳层。

阴影与模糊规则：

- 默认不加阴影，或只用极轻阴影。
- 模糊主要用于遮罩层和大型弹窗壳。
- 不允许夸张的悬浮卡片、厚重景深和发光效果。

### 4.6 间距

- 优先使用紧凑的桌面端间距，而不是营销页式的大留白。
- 小间距用于控件簇。
- 中间距用于模块分隔。
- 更大的垂直间距只用于消息块之间或弹窗大分区之间。
- 主应用区域不要出现巨大的 hero 式空白。

如果某个组件需要更强强调，应先提升对比、边框处理或字重，而不是先加颜色。

## 5. 排版规则

排版需要同时支持长时间阅读和工具型界面的密度感，但不能显得拥挤。

### 5.1 正文

- 默认正文大致在 `14px`。
- 正文行高应保证长文本阅读舒适。
- 无论是自然语言还是偏代码内容，都应保持可读密度。

### 5.2 标题层级

当前 markdown 标题尺度是基线：

- `h1`: `22px`
- `h2`: `18px`
- `h3`: `16px`
- `h4`: `15px`

规则：

- 标题应保持紧凑和文档感。
- 不应演变成超大号编辑型标题。
- 层级主要来自字重和间距，而不是强烈颜色。

### 5.3 字体使用

- 大多数 UI 文本使用正文字体。
- 标题、对话框标题等结构性文本使用 display 字体栈。
- 代码、命令样式文本、模型 ID 等偏机器感文本使用等宽字体。

## 6. 布局规则

### 6.1 应用壳层

- 应用整体是全高桌面布局。
- 根布局为双列结构。
- 左侧是固定宽度的会话侧栏。
- 右侧是自适应的聊天主区。
- 整体直接使用页面背景，而不是在最外层再包一个巨型外框面板。

不要把应用改造成：

- 带 hero 区块的营销首页
- 一堆分离卡片组成的 dashboard
- 除非产品需求明确改变，否则不要改成密集多窗格 IDE 布局

### 6.2 顶栏

- 顶栏应保持极简和工具化。
- 主要承载主题切换、设置入口和桌面窗口控制。
- 视觉上应安静。
- 不应变成品牌导航栏。

### 6.3 侧栏

- 侧栏应当窄、常驻、功能导向。
- 它使用平背景，而不是高抬升面板。
- 目录分组和线程行是主要信息结构。
- 选中和 hover 通过克制的背景变化表达。

### 6.4 聊天列

- 会话区域使用居中的阅读列。
- 内容宽度应保持紧凑且稳定。
- 整体更像聊天阅读器，而不是宽文档编辑器。
- 欢迎态和会话态应共享同一套居中内容逻辑。

### 6.5 Composer 的位置

- 在会话态中，composer 位于可滚动消息区下方。
- 它应共享与聊天列一致的横向内边距和宽度逻辑。
- 在欢迎态中，它属于中央空状态结构的一部分，而不是固定贴底。

### 6.6 对话框

- 设置弹窗是大型工作面板，并带内部滚动。
- 审批弹窗是更小的确认型对话框。
- 遮罩层允许模糊背景。
- 对话框表面本身应保持克制、可读和中性。

## 7. 组件规则

### 7.1 共享基础组件规则

共享 UI 基础组件是未来工作的默认起点。

优先复用：

- button
- input
- textarea
- select
- combobox
- scroll area
- overlay

除非共享基础组件确实无法承载，否则不要在功能代码里另起一套视觉实现。

### 7.2 按钮

- 按钮整体圆润且紧凑。
- 主按钮通过对比反转来建立强调，而不是靠品牌色。
- 次按钮和 ghost 按钮应保持安静。
- hover 强调主要来自轻微背景和文字变化。
- 小图标按钮要保持在圆形或接近圆形容器中的视觉平衡。

### 7.3 输入框与文本域

- 表单控件继承应用的字体体系。
- 控件整体应保持轻量、中性。
- 边框保持克制。
- 强蓝色 focus ring 不属于当前视觉体系。

composer 中的 textarea 规则：

- 它应当像嵌入在 composer 壳层里的一部分。
- 不应像“卡片套卡片”那样又重又独立。

### 7.4 Select、Combobox 与 Popover

- 浮层类选择面板使用弱化背景、柔和边框和适中的圆角。
- 选项高亮通过克制的填充变化实现。
- 带搜索的选择器也必须保持同样的黑白灰控件语言。
- 菜单和弹层应感觉紧凑高效，而不是宽松装饰化。

### 7.5 会话列表

- 目录分组使用小字号、低对比标签。
- 线程行保持紧凑。
- 活跃态通过克制的背景高亮表达。
- 删除等危险动作通常应放在 hover、focus 或二级菜单中出现。

### 7.6 Composer

- Composer 是界面中少数几个较大圆角表面之一。
- 它应把文本输入、附件、执行模式、模型选择、推理强度和发送或停止动作组织成一个整体。
- 它应感觉高密度、高效率，而不是大而装饰性强。

### 7.7 设置面板

- 设置应像一个统一的工具工作面板。
- 分区主要依靠间距和轻分隔线，而不是堆成一堵卡片墙。
- 服务列表与右侧详情区应读作同一个工作面。

### 7.8 审批弹窗

- 审批弹窗采用紧凑的确认交互模式。
- 风险信息和命令预览必须清晰、结构化。
- 但整体不应戏剧化，也不应被过度颜色编码。

## 8. 内容渲染规则

### 8.1 用户消息

- 用户消息是右对齐的紧凑气泡。
- 它应比常见消费级聊天气泡更轻。
- 使用克制的底色、较紧的内边距和中等圆角。
- 复制等辅助动作可以在 hover 或 focus 时再显现。

### 8.2 助手消息

- 助手输出应更像流式正文，而不是大边框响应卡片。
- 主要呈现载体是排版和间距。
- 常规助手输出不应被不必要地包成厚卡片。
- 错误信息在需要时可以使用轻容器。

### 8.3 Markdown

Markdown 应保持紧凑、清晰，并与整体 UI 排版一致。

规则：

- 标题保持克制和文档感。
- 段落间距保持可读。
- 行内代码和代码块使用中性样式。
- 不要做炫目的语法高亮主题秀。
- 表格、引用和链接都应易读，但不要变得过重。

### 8.4 工具输出与时间线项

- Thinking block、tool card 和时间线项可以使用容器。
- 但这些容器必须继续遵循同一套低对比黑白灰表面语言。
- 工具输出属于会话的一部分，而不是独立 dashboard。

### 8.5 流式输出

- 流式文本应以可读性和连续性为优先。
- 动效应感觉与内容到达节奏相关，而不是装饰性动画。
- Shimmer 只在思考中、标题生成中这类少数状态下有限使用。
- 不应把 shimmer 变成产品中随处可见的视觉母题。

### 8.6 空状态与欢迎态

- 欢迎内容应短、居中、功能导向。
- 它的职责是引导用户开始操作。
- 不应演变成品牌 hero 页面。

不要：

- 默认把每条助手消息都包成厚卡片
- 把工具输出改成彩色 dashboard 瓷砖
- 把欢迎态扩张成首页式大布局

## 9. 交互、动效与可访问性

### 9.1 交互

- Hover 状态应克制。
- 大部分 hover 反馈应来自中性背景或前景颜色变化。
- Focus 状态必须可见。
- Focus 的表现应保持平静，不要发光过猛。
- 次要动作可以在 hover 时出现，但关键动作必须仍能被键盘触达。

### 9.2 动效

动效应短、实用、低戏剧性。

当前动效词汇包括：

- fade-up 入场
- collapsible 展开和收起
- loading 旋转
- 少量用于生成相关状态的 shimmer

规则：

- 在引入新动画之前，优先复用已有这套动效语言。
- 动效应该服务于状态变化和可读性，而不是装饰。

### 9.3 Reduced Motion

- 必须尊重 `prefers-reduced-motion`。
- 流式输出和 shimmer 一类效果必须优雅降级。
- 即使去掉动画，状态表达也必须保持清晰。

### 9.4 可访问性

- 保持语义化按钮、对话框、标签和键盘交互模式。
- 不要把重要信息只藏在 hover 里。
- 在克制的灰阶视觉体系里仍要保证足够可读对比。
- 长文本聊天内容必须能支撑长时间阅读。

### 9.5 滚动行为

- 滚动区域视觉上应尽量轻。
- 当实际可导航性不受影响时，弱化或隐藏滚动条是可以接受的。
- 长对话与长设置面板应在合适位置独立滚动。

## 10. 响应式与运行时说明

- 这是一个桌面优先产品。
- 主要运行目标是 Tauri 桌面应用。
- 浏览器中的渲染需要继续可用于开发。
- 但界面语言应仍然体现桌面产品，而不是公开网站。

响应式规则：

- 第一优先级是内容连续性。
- 第二优先级是控件可用性。
- 除非产品需求明确改变，不要把它重塑成移动优先壳层。
- 在窄宽度下应做简化，而不是做装饰性适配。

## 11. Do / Don’t

### Do

- 使用黑白灰或近黑白灰作为视觉基础。
- 保持深色模式作为主要参考模式。
- 让层级主要由间距、对比和字重承担。
- 让聊天列保持紧凑和可读。
- 复用共享基础组件和共享 token。
- 保持桌面工具软件的整体气质。

### Don’t

- 不要引入大面积高饱和品牌色。
- 不要把应用做成营销首页。
- 不要构建一堵厚重的助手消息卡片墙。
- 不要加入戏剧性的阴影、发光或霓虹边缘。
- 不要在单个功能中引入和整体无关的新视觉系统。
- 不要让 hover、focus、active 反馈过猛。

## 12. 后续工作的判断标准

以后在这个仓库里新增或修改 UI 时，默认应该先问：

“这看起来像不像当前的 Prism-Agent 桌面客户端的一部分？”

如果一个改动必须依赖更亮的颜色、更重的包裹、更吵的动画，或者完全不同的布局模型，通常就说明它已经开始偏离当前设计系统了。
